# план реализации календаря питания и адаптивной готовки

> для выполнения нужен навык `superpowers:test-driven-development`; после каждого блока проводится проверка, а перед публикацией обновления pr используется `superpowers:verification-before-completion`.

**цель:** заменить перегруженный экран 14 дней на недельный календарь с устойчивыми партиями блюд и сделать инструкции готовки динамическими: они пересобираются после замен, учитывают реальную кухню, не конфликтуют по ресурсам и остаются полезными без openrouter.

**архитектура:** существующие `nutrition-core.js`, `nutrition-data.js` и `nutrition.js` остаются источником рациона и контроллером страницы. новая предметная логика выносится в три независимых umd-модуля: подготовка данных и калибровка, детерминированный планировщик ресурсов, клиент openrouter. интерфейс получает только проверенный локальным кодом план. модель декомпозирует рецепты, но не назначает окончательное время.

**стек:** vanilla javascript, html/css, `node:test`, localstorage через `Gamification.storeSet`, openrouter chat completions с `json_schema` и резервным `json_object`.

## общие ограничения

- не менять существующий формат `nutrition_state_v1` несовместимым образом;
- не отправлять в openrouter ключ, заметки пользователя, здоровье и другие данные вне состава недели;
- не синхронизировать и не экспортировать `openrouter_settings_v1`;
- не отображать поля ответа модели через `innerHTML`;
- не запускать два активных запроса для одного состава недели;
- не допускать пересечения активных действий пользователя, приборов, посуды и розеток;
- пассивные физические минимумы не уменьшаются калибровкой;
- на мобильном календарь не создаёт горизонтальную прокрутку;
- текстовые метки партий обязательны вместе с цветом;
- каждый завершённый блок фиксируется отдельным коммитом с сообщением в нижнем регистре на русском языке.

- не создаёт конфликтующих параллельных действий;
- применяет множитель `1.5` к активным действиям первой сессии;
- не уменьшает указанную пассивную варку или запекание.

### шаг 4: реализовать резервный план

если инструкции не содержат длительности, использовать консервативный диапазон на основе `prepMinutes`, делённый между шагами. помечать источник `fallback`, а предупреждение объясняет, что шаги последовательные и могут быть уточнены через api.

## задача 3: детерминированный планировщик ресурсов

**файлы:**

- создать `nutrition-scheduler.js`;
- создать `test/nutrition-scheduler.test.cjs`.

### шаг 1: написать тесты топологического расписания

проверить:

- зависимое действие начинается после завершения зависимости;
- два активных действия не занимают пользователя одновременно;
- пассивная варка может идти параллельно активной нарезке;
- одна кастрюля, конфорка или розетка не назначается двум действиям одновременно;
- аэрогриль и мультиварка в режиме `extended` получают комнатные розетки;
- тот же набор в режиме `winter` строится дольше либо возвращает объяснимую нехватку ресурсов;
- одинаковый вход всегда даёт одинаковое расписание.

### шаг 2: реализовать list scheduling

экспортировать:

```js
scheduleActions(actions, profile)
findResourceConflicts(schedule)
```

алгоритм:

1. проверить граф через предметный модуль;
2. взять доступные вершины в устойчивом порядке `batchId`, `id`;
3. для каждой вершины найти самый ранний интервал после зависимостей;
4. перебрать конкретные приборы, посуду и розетки с учётом помещения;
5. активному действию занять дорожку `user`;
6. после переноса между помещениями добавить служебный активный шаг;
7. вернуть `schedule`, `estimate.minMinutes`, `estimate.maxMinutes`, `peakOutlets`, `warnings`.

`maxMinutes` рассчитывать с переходным запасом и текущими коэффициентами, но не менять физические минимумы пассивных действий.

### шаг 3: добавить проверки конфликтов

`findResourceConflicts` должен отдельно сообщать пересечение пользователя, прибора, посуды и конкретной розетки. тесты создают заведомо плохое расписание и ожидают четыре разных кода ошибки.

### шаг 4: проверка и коммит

```bash

git add nutrition-scheduler.js test/nutrition-scheduler.test.cjs
```

- предупреждение о зависимостях перед пропуском.

переход текущего шага длится не более 200 мс и отключается при `prefers-reduced-motion`.

### шаг 5: реализовать настройки кухни и ручную пересборку

смена режима сохраняет профиль, меняет hash и пересобирает неделю. кнопка очистки удаляет только кеш готовки после подтверждения, не затрагивая рацион, импортированные блюда и историю циклов.

## задача 7: выполнение сессии и персональная калибровка

**файлы:**

- изменить `nutrition-cooking-core.js`;
- изменить `nutrition.js`;
- изменить `test/nutrition-cooking-core.test.cjs`;
- изменить `test/nutrition-ui.test.cjs`.

### шаг 1: написать тесты журнала сессии

проверить чистые функции:

```js
startSession(plan, now)
startStep(session, actionId, now)
pauseStep(session, actionId, now)
completeStep(session, actionId, now)
skipStep(session, actionId, now)
```

журнал должен переживать сериализацию, не учитывать паузу как рабочее время и не позволять начать заблокированный зависимостями шаг.

### шаг 2: написать тесты калибровки

`updateCalibration(profile, completedSession)`:

- обновляет только активные категории;
- использует ограниченное сглаживание;
- держит коэффициенты в безопасных границах;
- игнорирует слишком короткие или незавершённые наблюдения;
- не изменяет пассивные минимумы;
- позволяет сбросить калибровку.

### шаг 3: реализовать журнал и восстановление ui

текущее состояние сессии хранить внутри `nutrition_cooking_plans_v1` отдельно от кеша планов. после перезагрузки восстановить текущий шаг, паузы и завершённые действия. завершение всей сессии обновляет калибровку, инвалидирует будущие хеши и сохраняет фактическую длительность.
