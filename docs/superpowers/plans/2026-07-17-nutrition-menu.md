# план реализации раздела «меню»

## результат

добавить самостоятельный раздел питания с двухнедельным рационом, расширяемым каталогом блюд, недельной закупкой и отдельными сессиями готовки. состояние должно работать через существующее локальное хранилище и синхронизацию, а интерфейс — через общую навигацию приложения.

связанная спецификация: `docs/specs/nutrition-meal-plan-section.md`.

связанная задача: `#31`.

## принципы реализации

- чистая предметная логика не зависит от dom и доступна из node-тестов;
- встроенные данные отделены от пользовательского состояния;
- сначала добавляются падающие регрессионные тесты, затем минимальный код для их прохождения;
- каждое изменение состояния проходит через `Gamification.storeSet`;
- интерфейс не создаёт второй слой синхронизации;
- разделы страницы используют один контроллер и hash-навигацию;
- все оценки в ui остаются ориентировочными.

## шаг 2. описать контракты предметной логики тестами

файлы:

- создать `test/nutrition-core.test.cjs`;
- создать `nutrition-core.js` после получения ожидаемого падения.

тесты должны зафиксировать:

1. `normalizeLocalDate(value)` принимает `yyyy-mm-dd` и возвращает безопасную локальную дату;
2. `getCycleDay(startDate, currentDate)` возвращает дни 1–14 и признак завершённого цикла;
3. `validateImportedCatalog(payload, seedMeals, existingMeals)` принимает только формат версии 1, отбрасывает невалидные блюда и конфликты идентификаторов;
4. `mergeCatalog(seedMeals, importedMeals)` не изменяет исходные массивы и не даёт пользовательским блюдам перезаписать встроенные;
5. `resolveDayPlan(template, state, dayNumber, catalog)` применяет замены и безопасно возвращается к базовому блюду при неизвестном идентификаторе;
6. `calculateDaySummary(dayPlan, catalog, trainingExtra)` суммирует диапазоны калорий, белка и стоимости;
7. `buildShoppingList(template, state, catalog, weekNumber)` агрегирует одинаковые ингредиенты по идентификатору и единице;
8. замена блюда меняет состав закупки только соответствующей недели;
9. `normalizeState(raw, today)` восстанавливает повреждённые коллекции и сохраняет поддерживаемые данные;
10. `startNextCycle(state, startDate, mode)` создаёт снимок истории и очищает отметки нового цикла.

красная проверка:

```bash
node --test test/nutrition-core.test.cjs
```

ожидаемый результат до реализации: тестовый файл не может загрузить `nutrition-core.js`.

## шаг 3. реализовать чистую предметную логику

файл: `nutrition-core.js`.

экспортировать единый объект `NutritionCore` для браузера и `module.exports` для node.

реализовать:

```js
normalizeLocalDate(value)
getLocalDateKey(date)
getCycleDay(startDate, currentDate)
normalizeRange(value)
validateMeal(meal)
validateImportedCatalog(payload, seedMeals, existingMeals)
mergeCatalog(seedMeals, importedMeals)
createDefaultState(todayKey)
normalizeState(rawState, todayKey)
resolveMealId(template, state, dayNumber, slot)
resolveDayPlan(template, state, dayNumber, catalog)
calculateDaySummary(dayPlan, catalog, trainingExtra)
buildShoppingList(template, state, catalog, weekNumber)
createCycleSnapshot(state, template, catalog)
startNextCycle(state, startDate, mode)
```

ограничения:

- не читать `localStorage` и не обращаться к dom;
- не изменять переданные объекты;
- использовать календарные ключи дат, а не разницу часов;
- возвращать объяснимые ошибки импорта с кодом и сообщением;
- сохранять неизвестные дополнительные поля состояния только там, где это не мешает нормализации версии 1.

зелёная проверка:

```bash
node --test test/nutrition-core.test.cjs
```

коммит: `добавить основу данных для рациона`.

## шаг 7. создать каркас и стили раздела

файл: `menu.html`.

структура:

```text
nutrition-shell
  nutrition-sidebar
    today / cycle / dishes / shopping / cooking / history
  nutrition-main
    nutrition-toolbar
    view-today
    view-cycle
    view-dishes
    view-shopping
    view-cooking
    view-history
```

на desktop использовать боковую колонку фиксированной ширины и свободную рабочую область. на экране до 760 пикселей скрывать боковую колонку и показывать `select` или компактное меню. на экране до 480 пикселей карточки приёмов пищи, показатели и действия складываются вертикально.

не создавать вложенные декоративные карточки. рамка применяется к отдельному приёму пищи, строке блюда, позиции закупки или сессии готовки. внутренние разделы остаются плоскими рабочими областями.
